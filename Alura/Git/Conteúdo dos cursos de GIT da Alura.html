Comandos Iniciais

  • Iniciar a pasta como um repositório do git
    1. Acessar o diretório onde o repositório pretende ser será iniciado
    1. Acessado o diretório, executar o comando para promover o diretório para um gerenciado pelo GIT.
      ## Acessa o diretório gerenciado pelo git localmente
$ cd /[caminho para o repositório local]

## Inicializar o diretório como sendo um repositório gerenciado pelo git
$ git init

      Ao executar este comando, o Git Bash passará a exibir o diretório com a informação do nome da branch selecionada para o diretório gerenciado.

      ⚠️Atenção: O nome da branch mudou de master para main (01/10/2020+) por razões politicas pois, em inglês, a palavra master conota um sentido de senhor de escravos ou algo assim, mas é possível alterar o nome da branch principal da seguinte forma:

      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Altera o nome da branch principal de master para main
$ git branch -m master main

## Seta a alteração no repositório indicando que a branch de origem é a "main”
$ git push -u origin main

  • Configurar o usuário que opera o repositório

    Com o git instalado, é possível e recomendável que seja configurado o usuário que está operando o repositório para que o Git reconheça quem fez determinado commit/push para que se mantenha um registro das alterações de modo a permitir o rastreio do responsável pelas mudanças submetidas ao servidor.

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]
## Configurar o usuário local no terminal
$ git config --local user.name "Ramon Franco Simão"
## Configurar o e-mail do usuário local
$ git config --local user.email "ramon.simao@gmail.com"
    • Troca do usuário logado no GitHub

      É possível o cenário em que um usuário logado no git necessite ser alterado por um outro e para isso é preciso fazer o logout deste usuário para trocar por outro da seguinte maneira:

      • Windows
        • Apagar as credenciais do GitHub acessando pelo caminho: Painel de Controle\Todos os Itens do Painel de Controle\Gerenciador de Credenciais.
        • Na tela que foi acessada, clicar em Credenciais do Windows e localizar a credencial do GitHub.
        • Remover a credencial do GitHub do Git na máquina.
        • Assim que for feito um novo acesso ao GitHub pelo terminal do git, será pedido um novo login. Assim é possível alterar para o novo usuário.

    Obs: Para mais informações do que é possíveis configurar no Git, veja clicando aqui.

  • Manipulando arquivos no repositório

    A seguir será apresentado comando básicos para manipular arquivos gerenciados pelo repositório do Git…

    Mas antes de seguirmos para estes comandos, é necessário entender o conceito das áreas por onde os arquivos em um repositório podem estar

    • Conceitos do repositório Git
      • HEAD

        A HEAD do repositório representa a referência que o repositório local está fazendo para alguma branch. Ou seja, assim que o repositório é clonado pelo clone ou inicializado pelo init, o git aponta o HEAD para a branch padrão que é a master/main. Essa referência da HEAD sempre será alterada quando uma outra branch for selecionada.

        É possível verificar o referenciamento da HEAD no repositório da seguinte maneira:

        1. Acessar o Working Directory do repositório local pelo git bash;
        1. Acessar a pasta oculta .git
          ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Acessa o diretório oculto presente no Working Directory onde o repositório foi clonado ou iniciado
$ cd .git
        1. Verificar se existe o arquivo HEAD na pasta
          ## Lista todos os diretórios e arquivos contidos em .git
$ ls -ltra

      • Tracked File

        Este conceito indica que um determinado arquivo dentro do diretório do repositório não é reconhecido pelo Git como um arquivo rastreável ao qual o Git tem condição de gerenciar. Estes são os arquivos que ainda não foram submetidos a nenhuma área do repositório que serão apresentadas a seguir.

      • Working Directory ou Working Tree

        Essa área representa um diretório específico na máquina local que está relacionado a um repositório Git (Local e/ou Remoto). E neste diretório em que os arquivos podem ter as alterações rastreadas e gerenciadas pelo git.

        É também nesse diretório onde pode-se incluir arquivos Untracked.

        Todos os arquivos serão carregados para esse diretório mesmo quando em uma diferente branch for selecionada. É efetivamente o diretório de trabalho onde será possível editar localmente o conteúdo de arquivos e pastas.

      • Staging Area ou Index

        É nesta área em que o Git marca todos os arquivos e suas alterações que foram realizadas localmente no Working Directory e que foram adicionadas aqui através do comando add. Aqui é uma área intermediária entre a versão do arquivo local e o repositório local.

        Adicionar os arquivos na Staging Area sinaliza ao Git que estes arquivos estão disponíveis para envio ao repositório local através do comando commit.

        Além disso, os possíveis estados dos nossos arquivos são explicados com detalhes clicando aqui.

      • Repositório Local

        Repositório local é o repositório clonado a partir de um repositório remoto ou até mesmo um novo repositório criado através do comando init que inicializará um diretório qualquer como um repositório no Git local.

        Todo o conteúdo commitado no diretório local deve ser enviado para o repositório remoto sempre que possível de modo a sincronizar os conteúdos destes repositório para garantir sempre que as versões não se percam ou se sobreponham. A forma de sincronizar os conteúdos desses repositório é feita pelo comando push que envia os commits locais para o repositório local e o comando pull que trás todos os commits do repositório remoto cujo não existam no repositório local. Esses comando, assim como todos citados nessa seção serão abordados com mais detalhes durante esse curso.

      • Repositório Remoto

        O repositório remoto é onde todos os arquivos são centralizados e costuma ficar em um servidor de arquivos dedicado mas que também pode ser criado em qualquer computador através do comando init —bare que será abordado com maiores detalhes posteriormente.

    Uma vez compreendido por onde o arquivo pode passar e seus respectivos status, veremos os comandos que transitam os arquivos e suas alterações desde sua criação até o envio/coleta no servidor que hospeda o Repositório Remoto.

    • Incluir novos arquivos para a área de rastreio do repositório (Tracked File)
      • Qualquer novo arquivo incluído no diretório iniciado e gerenciado pelo Git deverá ser sinalizado para que o Git passe a monitorar e gerencial qualquer futura alteração ou até mesmo remoção do mesmo.
      • Para isso deve se seguir o seguintes passos:
        1. Acessar o diretório do repositório Git;
        1. Incluir um novo arquivo neste diretório fará que o Git detecte que aquele arquivo ainda não está sendo gerenciado por ele
          1. Para ter certeza quais são os arquivos que o Git não reconhece como arquivos gerenciados, basta executar o comando:
            $ git status
On branch master

  (use "git add <file>..." to include in what will be committed)
        Teste.txt

nothing added to commit but untracked files present (use "git add" to track)
          1. Com a certeza de que o arquivo Teste.txt não está sendo rastreado, agora basta transformar este em um arquivo rastreado através do seguinte comando:
            ## Diz ao git que leve o arquivo para a área de stage
## É possível ter um efeito muito similar ao comando “git add” apenas colocando um “.” no lugar do nome do arquivo. 
## Mas é preciso saber que ao optar pelo uso do “.” fará com que todos os arquivos pendentes de adição serão adicionados todos de uma só vez.
$ git add Teste.txt
$ git status

  (use "git push" to publish your local commits)

  (use "git restore --staged <file>..." to unstage)
        new file:   Teste.txt
    • Promover os arquivos adicionados na Staged Area (local onde os tracked files são gerenciados) para o repositório Git

      Com todos os arquivos que deseja-se enviar para o repositório adicionados na Staging Area, é preciso submeter as alterações feitas no Working Director para o repositório local e para isso o seguinte comando é necessário:

      ## O comando commit demanda, através do parâmetro -m, que seja incluída uma mensagem de descrição do commit.
$ git commit -m "Mensagem para descrever o commit”
    • Como analisar o histórico de commits (log)?

      Para ter acesso ao histórico de commits é possível faze uso do comando log em conjunto de seus parâmetros opcionais. A seguir veremos como este comando funciona:

      Este comando traz informações que nos permitem identificar commits através de um hash, saber quem foi o responsável pelo commit e qual a data em que o commit foi realizado. Isso é muito útil para verificar modificações, encontrar onde um bug surgir por meio do comando bisect ou encontrar quem alterou uma linha específica em algum arquivo através do comando blame.

      O comando bisect e blame serão abordados com maior detalhe posteriormente.

        Ao executar este comando, o retorno mostra o seguinte

Repositório Remoto

O repositório local do git é onde os arquivos gerenciados por ele são centralizados e e ficam disponíveis para que outros colaboradores tenham acesso aos arquivos e contribuam com atualizações.

Um repositório remoto pode ser criado em diversos ambientes como repositórios de referência no mercado e até mesmo em servidores e máquinas pessoais que se conversam em um rede fechada.

Os exemplos de repositórios remotos são:

  • Repositórios de mercado
    • GitHub

      ⚠️ O GitHub lançou recentemente a opção de ter repositórios privados para projetos com até 3 colaboradores.

    • GitLab
    • Bitbucket

      ⚠️ O Bitbucket continua sendo uma saída gratuita interessante para projetos com mais colaboradores.

  • Repositório próprio

    Neste tipo de repositório local é possível criar em um servidor em uma rede própria onde a responsabilidade de manter este é de quem o cria e dos administradores da máquina ou da rede. A seguir abordarei com maiores detalhes como esse servidor remoto pode ser criado e/ou mantido.

  • Inicialização do repositório remoto (próprio)

    Com este comando nós criamos um repositório que não terá a Working Tree, ou seja, não conterá uma cópia dos nossos arquivos. Como o repositório servirá apenas como servidor, para que outros membros da equipe sincronizem seus trabalhos, poupamos espaço de armazenamento desta forma.

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd [Caminho onde se deseja criar o repositório remoto]
## Cria um diretório que será inicializado como repositório remoto na máquina
$ mkdir servidor
## Acessa o diretório criado
$ cd servidor
## Inicializa este diretório informando ao git que ele será um repositório remot
## Será criado nesse diretório todos os arquivos e pastas que o git necessita para  gerenciar o repositório.
$ git init --bare

    • Observações
      • Ao fim da execução deste comando, o diretório possuirá a marcação (BARE:master) no terminal para indicar que o diretório agora será reconhecido como um repositório remoto.
      • A palavra bare em inglês possui os seguintes significados (Nu, Vazio, Descoberto) no sentido de ser um repositório vazio ao qual servirá APENAS para conter e administrar os repositórios que ele conterá.
      • Caso se esqueça de escrever o parâmetro --bare no comando init, será iniciado um repositório local normal e para isso ser revertido é necessário executar o comando a seguir para dizer ao git transformar um repositório comum em um remoto.
        $ git config core.bare true
  • Apontar um repositório local para um repositório remoto

    Pode ser necessário apontar um repositório local para enviar e receber dados de um repositório remoto qualquer e para isso é necessário acessar o diretório do repositório local e nele executar os seguintes comandos que podem nos ajudar a listar, adicionar e remover o link com repositórios remotos de escolha.

    • Listar
      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]
## Lista os repositórios remotos que estão associados ao repositório local
$ git remote

    • Listar detalhadamente
      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]
## Com esse parâmetro é listado os repositórios locais incluíndo a informação do caminho do fetch e push
$ git remote -v

    • Adicionar
      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]
## Cria a referência entre o repositório locam com o remoto
$ git remote add [Nome do repositório] [caminho para o repositório remoto]
## Após a adição é possivel validar a associação listando os repositórios remotos
$ git remote

  • Clonando um repositório remoto

    Para trazer um repositório remoto para o nosso git local é necessário cloná-lo em uma pasta do sistema de arquivos do nosso sistema operacional que vamos trabalhar. Essa clonagem trará todo os arquivos gerenciados pelo repositório remoto bem como metadados como branches, commits, logs e etc (exceto os arquivos específicos de controle e gerenciamento do repositório remoto).

    ## Acessa o local no sistema de arquivos do SO onde o repositório remoto será clonado
$ cd [Diretório base da clonagem]
## Clona o repositório remoto no caminho atual no sistema de arquivos da máquina
## É possível dizer qual será o nome do repositío(que inclusive será criado uma pasta)
## Caso não seja informado o nome da pasta, o repositório remoto será clonado no diretório base
$ git clone [Caminho para o repositório remoto] repositorio
  • Enviando e recebendo dados de um repositório remoto
    • Enviando commits para o repositório remoto

      Os arquivos alterados no repositório local que foram promovidos ou adicionado na Staged Area e em seguida commitados só serão enviados para o repositório local. Para enviá-los ao repositório remoto é necessário “empurrar” todos os commits realizados no repositório local para o repositório remoto de onde os arquivos se originam.

      • Push (empurrar)
        ## Envia os commits que existem apenas no repositório local para o remoto
## O repositório remoto é indicado através do seu nome (vide o comando git remote add)
## É importante também informar qual a branch que os dados serão enviados lá no repositório remoto
## Se a branch for omitida do comando, corre-se o risco de que as alterações sejam enviadas para todos as branchs do repositório remoto
$ git push [Nome do repositório] [Nome da Branch]
    • Recebendo commits para o repositório local

      Caso o repositório remoto receba commits que não existiam antes de ser clonado na máquina local, pode ser necessário “puxar” esses commits para o repositório local de modo que ambos repositórios permaneçam em sincronia.

      • Pull (puxar)
        ## Recebe os commits que estejam presentes no repositório remoto mas não no local
## O repositório remoto é indicado através do seu nome (vide o comando git remote add)
## Se a branch não for informado, o git entende que é a branch index do repositório local que deverá ser buscadas as alterações no remoto
$ git pull [Nome do repositório] [Nome da Branch]

          1. git clone -> Realiza a cópia de um repositório remoto para uma pasta local, trazendo todo o histórico de mudanças,
            conhecidos como commits, e todos os arquivos
            desde o inicio.
          1. $ git remote add -> Permite interligar um repositório local já existente a um repositório remoto, assim podemos começar a enviar mudanças e commits para esse repositório remoto com o git push.
          1. $ git push -> Envia todas as alterações para o repositório remoto, isso inclui arquivos novos, mudanças em arquivos e até mensagens
            de commits.
          1. $ git pull -> Traz todas as alterações que estão presentes no repositório remoto, e se diferencia do $ git clone pelo fato de não criar um novo repositório local que tem que trazer todo o conteúdo, trazendo apenas as alterações que o repositório local ainda não tem, acelerando o trabalho para repositórios muito grandes.

Trabalhando com Branches

As branches no git são estruturas que nos permitem ramificar um repositório em diversas versões sendo a branch configurada como upstream a ramo principal de onde as demais devem variar e sincronizar os dados de modo a centralizar tudo em uma branch principal. Que anteriormente nesse manual criamos com o nome de master.

Ao acessar localmente uma repositório pelo GitBash, o terminal mostrará para qual branch este diretório está apontando.

## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Ao acessar o caminho do repositório local clonado do remoto, a seguinte estrutura aparecerá no terminal
[USUÁRIO]@[NOME DA MÁQUINA] MINGW64 /[UNIDADE DE DISCO]/[CAMINHO]/[REPOSITÓRIO] (master)
  • Manipulando Branches
    • Branch

      Branches ("ramos") são utilizados para desenvolver funcionalidades isoladas umas das outras. A branch master é a branch "padrão" quando você cria um repositório.

      É interessante separar o desenvolvimento de funcionalidades em branches diferentes, para que as mudanças no código, para um ramo, não influencie no funcionamento de outro.

      Alguns comandos principais são:

      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Cria uma branch chamada [Nome da Branch]
$ git branch [Nome da Branch]

## Lista todas as branches existentes para o repositório
$ git branch

    • Checkout

      Usado para deixar o nosso código em determinado estado (navegar entre branches, fazer o restore de versões na Working Tree e das que estão em staged).

      A descrição do comando git checkout --help, em uma tradução livre é: "Atualizar os arquivos na working tree para ficarem na versão especificada. [...]". Basicamente, podemos deixar o nosso código no estado do último commit de uma branch, de um commit específico, ou mesmo tags (que veremos adiante).

      Alguns comandos principais são:

      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Indica ao git que é desejado que a branch index seja alterada para [Nome da Branch] 
$ git checkout [Nome da Branch] 

## O parâmetro -b informa ao git para trocar para [Nome da Branch] e, caso ela não exista, então o git criará-la antes da troca.
git checkout - b [Nome da Branch]
      • Observações
        • 🗒️ NOTA: Ao se alterar de uma branch para outra, as alterações que foram commitadas em cada branch são trazidas para os arquivos após a mudança da branch.
        • ⚠️ ATENÇÃO: Caso as alterações feitas em uma branch não tenham sido commitadas antes da alteração entre as branches, todo trabalho não commitado será perdido.
  • Integrando diferentes alterações em diferentes branches

    Como dito anteriormente, as branches se relacionam entre si e por muitas vezes é importante que elas sejam sincronizadas de modo a levar alterações de uma para a outra mantenho suas características de versões únicas.

    Imagine que exista um arquivo na branch master que não foi alterado por nenhum usuário até o momento:

    Um exemplo de arquivo

    Texto.txt - [BRANCH 1]

      E então temos os seguintes cenários…

      1. Imagine que temos um arquivo chamado Texto.txt na branch master e o mesmo arquivo está na branch teste. O mesmo arquivo foi alterados da seguinte maneira em cada branch

        Texto.txt - [BRANCH 1]

        Um exemplo de arquivo que foi alterado na branch BRANCH 1

        Texto.txt - [BRANCH 2]

        Um exemplo de arquivo que foi alterado na branch BRANCH 1
Além de ser alterado tambem na branch BRANCH 2

        Perceba que o mesmo arquivo foi alterado mas em regiões do texto que são destintas e que não entram em conflito entre as duas branches.

      1. Imagine que temos um arquivo chamado Texto.txt na branch master e o mesmo arquivo está na branch teste.

        Texto.txt - [BRANCH 1]

        Um exemplo de arquivo que foi alterado na branch BRANCH 1

        Texto.txt - [BRANCH 2]

    • Comando Merge

      O conceito de merge significa que uma branch pode ser mesclada a outra, isto é, trazer de um branch as alterações nela feitas para os arquivos e pastas da branch que receberá essas alterações.

      O merge é mais indicado para trabalhos com branches compartilhadas, onde há várias pessoas envolvidas alterando diversas branches;

      Alguns comandos principais são:

      • Cenários 1
        ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local] 

## Muda para a branch que receberá as alterações de outra branch.
$ git chekout [BRANCH 1] 

## Solicito ao git para mesclar tudo que foi alterado na [BRANCH 2] com a [BRANCH 1] 
$ git merge [BRANCH 2] 

## O git cria um nove commit na branch [BRANCH 1] com as alterações recebidas da branch [BRANCH 2].
$ git log --oneline
      • Cenários 2
        ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local] 

## Muda para a branch que receberá as alterações de outra branch.
$ git chekout [BRANCH 1] 

## Solicito ao git para mesclar tudo que foi alterado na [BRANCH 2] com a [BRANCH 1] 
$ git merge [BRANCH 2]

        Como neste cenário, o mesmo trecho do arquivo foi modificado em ambas as branches que se deseja mesclar, isso acaba gerando um conflito que precisa ser tratado. Então o git adiciona o conflito no arquivo para ser tratado da seguinte maneira:

        Texto.txt - [BRANCH 1]

        >>>> HEAD
Um exemplo de arquivo que foi alterado na branch master
>>>> [HASH DO COMMIT FEITO PELA BRANCH 2]
Uma exemplo de arquivo que foi alterado nas branches BRANCH 1 e BRANCH 2

        Então basta, manualmente, resolver o conflito

        Um exemplo de arquivo que foi alterado nas branches BRANCH 1 e BRANCH 2

        E agora enviar as alterações mescladas para a branch que se pretende receber as alterações

        ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local] 

## Adiciona o arquivo alterado à Staged Area
$ git add Texto.txt

## Commita a alteração para a branch BRENCH 1
$ git commit -m "Realizado o tratamento de conflito do merge"

## "Empurra" todos os commits para o repositório remoto (GitHub)
## Lembrando que origin é o nome do repositório remoto
$ git push origin [BRANCH 1]

    • Comando Rebase

      A função do comando rebase se assemelha um pouco com o merge mas com a diferença que ao invés da mesclagem dos commits, aqui se mescla as branches em si. Ou seja, é possível fazer uma fusão de dois branches reduzindo todo o histórico de commits das duas branches em um único commit.

      ⚠️Atenção: O uso do comando rebase assim como o comando merge pode detectar conflitos entre as versões das branches (como no Cenário 2) e portanto a mesma tratativa se aplicará.

      O rebase é mais indicado para projetos pessoais, times pequenos ou projetos mais simples.

      Utilizando a ferramenta Visualizing Git, vamos a um exemplo prático

      ## Aponda a HEAD do repositório para a branch master 
$ git checkout master

## Executa uma série de commits na branch master 
$ git commit -m "C0"
$ git commit -m "C1"
$ git commit -m "C2"

## Cria e aponta a HEAD para a nova branch experiment 
## Essa branch é criada apartir do versão correspondente ao commit "C2"
$ git checkout -b experiment

## Retorna a HEAD do repositório para a branch master 
$ git checkout master

## Faz um novo commit na master 
## Isso cria agora uma diferença entre a versão da master e da experiment 
$ git commit -m "C3"

## Retorna a HEAD do repositório para a branch experiment 
$ git checkout experiment

## Faz um novo commit na experiment 
## Isso cria mais uma diferença entre a versão da master e da experiment 
$ git commit -m "C4"

      Ao fim da execução de todos esses comando acima, teremos a seguinte representação visual:

      Note que a branch master e a experiment derivam da mesma versão ”C2” e a partir dela se diferem (onde a master agora tem a versão “C3” e a experiment tem a versão “C4").

      Então neste momento seria possível optar pelo merge que foi apresentado anteriormente que iria mesclar todas os commits da experiment na master e também é possível mesclar todos os commits das duas branches em um só.

      ## Retorna a HEAD do repositório para a branch experiment 
git checkout experiment

## Ao executar este comando
## O Git fará a mescla da versão C3 na master na versão C4 da experiment 
## Um novo commit na experiment é realizado levando todas as alterações de ambas as branches
## A versão C4 que existia apenas na experiment é abandonada
## Neste momento a branch experiment detém a mescla das versões C3 e C4
## Já a branch master master, apenas detém a versão C3
$ git rebase master

      Agora basta mesclar a versão da experiment na master para equalizar as duas branches

      ## Retorna a HEAD do repositório para a branch master 
$ git checkout master

## Mescla a versão da master(C3) com a experiment(C3+C4)
$ git merge experiment

Versionamento

O versionamento é uma estratégia do git que nos permite manipular versões, “viajar no tempo” para versões anteriores em relação a nossa Working Tree salvar alterações em espaços temporários no repositório, gerar Tags e Releases , fazer uma varredura entre os commits afim de encontrar um exato ponto onde uma alteração específica ocorreu (Cherry Pick) e encontrar o responsável por alguma alteração em trechos dos arquivos gerenciados pelo Git (Blame).

  • Desfazendo ou recuperando modificações

    Em algumas ocasiões pode ser interessante retornar a versão de algum arquivo na Working Tree para a versão da HEAD e para isso é utilizado o comando que já vimos anteriormente neste arquivo mas seu uso é um pouco diferente como demonstrado a seguir:

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Com este comando estamos indicando ao git que desfaça as alterações do arquivo na Working Tree.
$ git checkout -- [ARQUIVO]

    Também é possível obter o mesmo efeito da seguinte maneira:

    ## Acessa o diretório gerenciado pelo git localmente
$ cd /[caminho para o repositório local]

## Com este comando estamos indicando ao git que desfaça as alterações do arquivo na Working Tree.
$ git restore [ARQUIVO]

  • Restaurando alterações que já foram adicionadas para área HEAD / Stage Area

    Outra necessidade pode estar em remover arquivos ou alterações que foram enviadas para a HEAD ou para a Staged Area e para tanto usamos os seguintes comandos:

    • Remoção da HEAD (reset)
      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Com este comando estamos indicando ao git que remova o arquivo da HEAD.
$ git reset HEAD [ARQUIVO]
    • Remoção da Staged Area (restore)
      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Indica que a restauração se aplica à Staged Area 
$ git restore --staged [ARQUIVO]
  • Removendo commits indesejados (revert)

    É possível que também se deseje remover commits que foram feitos equivocadamente ou que tenham sido indesejados para aquele momento em particular e é isso que será demonstrado abaixo:

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local] 

## Lista os commits realizados na branch com a inteção de localizar o que se deseja a remoção
$ git log
commit [HASH do Commit] (HEAD -> [BRANCH para onde o commit foi empurrado])
Author: [Nome do Autor] <[E-mail do Autor]>
Date:   [Data do Commit]

[Mensagem do Commit] 

## Uma vez obtido o hash do commit, basta informalo no comando para desfazer as alterações
$ git revert [HASH do Commit]
  • Viajar no tempo entre os commits (checkout)

    Outro recurso que o git oferece é o de baixar para a Working Tree a versão de um determinado commit de modo a voltar no tempo em que aquela versão estava na HEAD da branch, veja como isso pode ser feito:

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local] 

## Lista os commits realizados na branch com a inteção de localizar o que se deseja a remoção
$ git log
commit [HASH do Commit] (HEAD -> [BRANCH para onde o commit foi empurrado])
Author: [Nome do Autor] <[E-mail do Autor]>
Date:   [Data do Commit]

## Uma vez obtido o hash do commit, basta informalo no comando para acessar a versão do commit
## É importante notar que não estamos alterando ou desfazendo nada, apenas acessando a versão do commit
## Apesar de não haver alterações, certifique-se de antes commitar qualquer alteração na Working Tree pois após a execução deste comando tudo o que não estiver commitado será perdido.
$ git checkout [HASH do Commit]
  • Salvando Temporariamente o conteúdo da Working Tree (stash)

    Em inglês, a palavra stash significa algo como esconder, juntar, acumular ou até mesmo esconderijo. O que o git stash oferece é justamente recolher todo o trabalho não enviado para a staged area presente na Working Tree e enviá-lo para um esconderijo temporário que poderá ser acessado novamente.

    Isso mostra-se muito útil quando é necessário modificar a HEAD da Working Tree para acessar outra versão dos arquivos mantendo as alterações não adicionadas na staged area disponíveis para quando for necessário recuperá-las e retomar o trabalho.

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Solicita ao git que leve todo o trabalho não submetido para a área stash (esconderijo)
$ git stash

## Lista tudas as alterações escondidas
$ git stash list

## Recupera para a Working Tree as alterações gravadas em uma stash pelo seu index
$ git stash apply [INDICE DA ALTERAÇÃO]

## Descarta do esconderijo a última alteração
$ git stash drop

## Seleciona através de um índice qual alteração deve ser descartada
$ git stash drop stash@{[INDICE DA ALTERAÇÃO]}

## Recupera e logo em seguida remove do stash a última alteração
$ git stash pop

## Seleciona através de um índice qual alteração deve ser recuperada e logo em seguida descartada
$ git stash pop stash@{[INDICE DA ALTERAÇÃO]}
  • Diferencias entre as versões (diff)

    Com o comando diff é possível analisar diferenças entre versões de commits e branches para ser possível avaliar quais mudanças ocorreram em um determinado momento e por um autor específico.

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Solicita ao git que informe somente as diferenças entre a HEAD e Working Tree
$ git diff

## Compara as diferenças entre o Commit 1 e Commit 2
## Os commits são informados usando apenas os 7 primeiros dígitos dos seus hashes
$ git diff [HASH COMMIT 1]..[HASH COMMIT 2]

## Compara as diferenças entre a Branch 1 e Branch 2
## Os commits são informados usando apenas os 7 primeiros dígitos dos seus hashes
$ git diff [HASH BRANCH 1]..[HASH BRANCH 2]
  • Gerar Versão (tags / release)

    O Git oferece também a possibilidade de definir marcos de versões de release e versões de deploy para produção.

    • Tag

      Uma Tag no Git é uma referência estática a um ponto específico na história do seu repositório. Ela é usada para marcar commits importantes, como versões estáveis, marcos de desenvolvimento ou releases principais. Basicamente, uma Tag é uma forma de nomear um commit específico para que seja possível referenciá-lo e identificá-lo facilmente no futuro.

      ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Lista todas as tag do repositório
$ git tag

## Cria uma tag atribuindo a ela um nome/versão e uma mensagem de descrição
$ git tag -a [Número da versão] -m "[Mensagem de descrição]"

## É muito importante que caso a tag seja criada pelo repositório local, a criação seja empurrada para o repositório remoto
$ git push origin [Número da versão]
    • Release

      Uma Release no GitHub é uma versão específica do seu projeto que é disponibilizada para download ou deploy. É uma forma de empacotar um conjunto de arquivos do seu repositório em um formato que possa ser facilmente distribuído. Uma Release geralmente contém arquivos compilados, binários, documentação ou qualquer outra coisa necessária para distribuir seu software.

      Ao criar uma Release no GitHub, você pode escolher quais commits, Tags ou branches deseja incluir no pacote. Isso permite que você selecione exatamente quais alterações ou versões do seu código fazem parte daquela Release específica. Cada Release geralmente possui um número de versão associado, como v1.0.0, v2.1.3, etc.

      Ao enviar uma tag para o GitHub, via comando git push, uma release não é criada automaticamente. Caso você queira criar uma release baseada em uma tag,
      deve fazer isso manualmente. A seguir um passo a passo de como realizar esse procedimento:

      1. Acesse a página do seu repositório no GitHub e clique no link Tags;
      1. Na página de listagem de tags, clique no botão Releases;
      1. Repare que o GitHub indica que não existem releases no repositório e exibe um botão para criar uma nova release. Clique nesse botão para abrir a página de criação de release, na qual você deve escolher a tag associada à nova release sendo criada, além de também poder preencher um título e uma descrição;
  • Selecionar uma versão de commit específica (cherry-pick)

    Existe uma expressão em inglês que seria o equivalente à expressão “Escolher a dedo” em português que é “Cherry Pick”. Essa expressão se encaixa muito bem com o comando de selecionar a versão de uma branch específica e trazê-la pra a branch da HEAD.

    Digamos que tenhamos o seguinte cenário onde a Branch_1 e Branch_2 estão em linhas distintas de evolução a partir de um ponto específico da sua linha do tempo de commits

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local]

## Seleciona a branch onde haverá uma linha de evolução da versão
$ git checkout -b Branch_1
$ git commit -m "B1 1"
$ git commit -m "B1 2"

# Uma nova branch é criada e iniciá-se uma nova linha de evolução paralela ao master
$ git checkout -b Branch_2
$ git commit -m "B2 1"
$ git commit -m "Levar para a Branch_1"
$ git commit -m "B2 2"

## Retorna a HEAD para a master e novamente atualiza sua versão 
$ git checkout Branch_1
$ git commit -m "B1 3"

    Então imagine que seja interessante para a evolução da Branch_1 possuir em um determinado momento uma alteração específica feita na Branch_2 em algum momento de sua evolução. Podemos imaginar que no commit b46a1b0 foi encontrado algum bug que foi corrigido mas que precisa ser enviado apenas este para a branch Branch_1 sem necessariamente ter que atualizar as duas branches. É ai que utilizamos o cherry-pick que trará à Branch_1 apenas a versão necessária que se encontra no commit b46a1b0.

    ## Traz a versão contida no commit b46a1b0 submetido pela Branch_2 para a Branch_1 
$ git cherry-pick b46a1b0

  • Encontrando um bug(biserct)

    Biserct é uma maneira que o git disponibiliza para localizar dentro da linha de evolução de uma branch qual foi o momento evolucionário que uma característica ou bug foi inserido. 

    ## O bisect solicita que você informe, ao ser inicializado, um commit que sabemos que não tem o bug e outro commit que sabemos que o bug está ocorrendo
$ git biserct start

## Uma vez inicializado é necessário informar para o GIT qual o ponto de partida da pesquisa (Ponto em que o código não está como o esperado)
$ git bisect bad HEAD

## Agora é preciso informar para o GIT qual o ponto do versionamento do código uma alteração que agora está quebrada mas em uma determinada versão estava OK
$ git bisect good [HASH DO COMMIT] 

## Com todos os parâmetros da busca estabelecidos, é possível trazer as alterações para o HEAD ao informar ao GIT se a revisão é "good" ou "bad"
$ git bisect bad
$ git bisect good
  • Encontrando os responsáveis por uma alteração (Culpado)

    Este comando serve para identificar o usuário que submeteu (commit) cada modificação em um determinado arquivo de modo a permitir buscar com o responsável alguma orientação ou esclarecer alguma dúvida referente a modificação.

    Outra característica interessante para este comando é que é possível verificar a data que a modificação foi realizada.

    $ git blame index.html

    Por exemplo

    Imagine que exista um arquivo chamado Culpado.txt que possua o seguinte conteúdo:

    Este arquivo nasceu assim
Permanecerá assim
até que alguém o modifique

    Então dois autores distintos fazem alterações e as submetem para o repositório

    Autor 1

    Este arquivo nasceu assim
Permaneceu assim
até que o Autor 1 o modificou

    Autor 2

    Este arquivo nasceu assim
Permaneceu assim
até que o Autor 1 o modificou
e logo em seguida o Autor 2 também contribuiu

    Ao executar o comando git blame Culpado.txt será exibido no terminal algo parecido com isso

    [HASH COMMIT 1] (Autor 0 [DATA DO COMMIT] [LINHA]) Este arquivo nasceu assim
[HASH COMMIT 2] (Autor 1 [DATA DO COMMIT] [LINHA]) Permaneceu assim
[HASH COMMIT 2] (Autor 1 [DATA DO COMMIT] [LINHA]) até que o Autor 1 o modificou
[HASH COMMIT 3] (Autor 2 [DATA DO COMMIT] [LINHA]) e logo em seguida o Autor 2 também contribuiu

  • Eventos

    Dentro do diretório do git na pasta do repositório clonado é possível inserir scripts (.sh) de eventos a serem executados pelo git. Para isso basta acessar o diretório de eventos do repositório (hooks) e editar o arquivo referente ao evento que seja interessante embutir algum comportamento.

    ## Acessa o diretório gerenciado pelo git localmente (caso não esteja nela ainda)
$ cd /[caminho para o repositório local] 

## Acessa o diretório de hooks(ganchos) para os eventos no git para o repositório
$ cd .git/hooks

## Ao listar os arquivos do diretório de hooks, é possível ver as possibilidades de eventos
$ cd ls -ltra

    Dentro do diretório de hooks é possível criar um arquivo com o nome do evento (Mesmo nome dos arquivos existentes no diretório mas sem o .sample no final)

    ## O arquivo será criado e o editor "vim" o abrirá para edição
$ vim pre-commit

    Ao abrir o editor, é necessário incluir o comando #/bin/sh na primeira linha do arquiva para indicar que o conteúdo do arquivo se trata de um script Linux. Por exemplo:

    $ #/bin/sh
$ echo "Este texto irá aparecer sempre antes de executar um commit para o reposítório"

  • Git Flow

    O Git Flow é um modelo de fluxo de trabalho para o Git, um sistema de controle de versão distribuído. Foi criado por Vincent Driessen em 2010 e desde então tem sido adotado por muitos desenvolvedores de software em todo o mundo. A ideia principal do Git Flow é ter um fluxo de trabalho estruturado que seja adequado para projetos com lançamentos regulares. Ele oferece um modelo robusto de gerenciamento de ramos para diferentes versões, recursos e correções de bugs.
