O que são APIs

API vem do inglês e significa “Interface Programming Interfaces” e na prática é um conjunto de funções e características de um programa que permite a integração de outro programa que não tem o intuito de participar dos detalhes de implementação interna.

APIS Comuns
  • APIs Web

    Exemplo: A API do google que fornece um endereções a partir de uma latitude e longitude.

    • Padrões
      • RPC
        • Remote Procedure Call - Informava na URL o nome do recurso que se pretende executar no servidor.
      • SOAP
        • Protocolo para trabalhar com o padrão RPC baseado em XML.
      • RESTful
        • Utiliza o protocolo HTTP e seus verbos para chamar procedimentos remotamente em um servidor.
      Outras Possibilidades
      • Vantagem
        • Usado para criar APIs de acesso mais mais flexível aos dados expostos por ela já que ele funciona quase como uma query.
      • Desvantagem
        • Complexo.
        • Transfere a responsabilidade do uso para o usuário ou cliente (exigindo que eles tenham o conhecimento básico sobre este padrão).
      • Vantagem
        • Criar APIs com um foco maior em performance por conto de estratégias de compressão(por padrão) e baixa latência.
      • Desvantagem
        • Complexo de Implementar.
  • APIs de código fonte

    Exemplo: A API que fornece acesso à logica de tratamento de datas no java.

CORS (Cross-origin Resource Sharing)

Ocorre quando uma aplicação que está em outro domínio tenta ser acessada de alguma maneira não permitida pela política de segurança CORS no Backend. Para impedir que problemas como esses ocorram, é necessário que seja adicionado o cabeçalho permitindo a ação da requisição no backend no back-end para permitir que o domínio não sofra com a política de CORS.

Essa política é aplicada apenas pelo navegador e não é algo ligado ao protocolo HTTP, então não se espera que isso aconteça com outros HTTP Clients que não sejam navegadores web.

Conceitos

Métodos / Verbos HTTP (RFC 7231 e RFC 5789)

GET (RFC 7231)

Obtém um único recurso ou um grupo deles ao servidor

Propriedades Comuns:

  • ❌ Requisição necessita de um body.
  • ✅ Resposta com body.
  • ✅ Safe Method - Pois o a consulta por si só não altera o estado do recurso no servidor.
  • ✅ Idempotente - Pois a múltiplas consultas ao recurso se manterá a mesma independente de quantas requisições repetidas sejam enviadas.
  • ✅ Cacheable Method.

HEAD (RFC 7231)

Consulta no servidor informações de metadados a respeito do endpoint.

Propriedades Comuns:

  • ❌ Requisição necessita de um body.
  • ❌ Resposta com body.
  • ✅ Safe Method - Pois o a consulta por si só não altera o estado do recurso no servidor.
  • ✅ Idempotente - Pois a múltiplas consultas ao recurso se manterá a mesma independente de quantas requisições repetidas sejam enviadas.
  • ✅ Cacheable Method.

POST (RFC 7231)

Cria um novo recurso no servidor

Propriedades Comuns:

  • ✅ Requisição necessita de um body.
  • ✅ Resposta com body.
  • ❌ Safe Method - Pois este método tem justamente a função de alterar recursos ou estados no servidor.
  • ❌ Idempotente - Pois múltiplas requests deste verbo podem multiplicar o mesmo recurso no servidor (por exemplo, adicionar o mesmo curso múltiplas vezes).
  • ⚠️ Cacheable Method - Cacheável somente se a resposta tiver definido explicitamente informações de quão recente é a informação (freshness) e um cabeçalho Content-Location for definido para o mesmo valor da URL acessada.

    Informações de freshness podem ser fornecidas através de cabeçalhos como Expires ou Cache-Control: max-age.

PUT (RFC 7231)

Atualiza de uma só vez todo um dado recurso no servidor.

Propriedades Comuns:

  • ✅ Requisição necessita de um body.
  • ✅ Resposta com body.
  • ❌ Safe Method - Pois os recursos enviados serão alterados múltiplas vezes no servidor mesmo que a alteração seja sempre para os mesmos dados.
  • ✅ Idempotente - Pois múltiplas requests deste verbo afetaram o recurso com o processamento da alteração mas não deve fazer diferença para o estado do recurso no servidor já que os dados permanecem o mesmo.
  • ❌ Cacheable Method.

PATCH (RFC 5789)

Semelhante ao PUT mas deve ser usado apenas para atualizar um recurso de forma parcial… Por exemplo, atualizar apenas uma propriedade do recurso.

Propriedades Comuns:

  • ✅ Requisição necessita de um body.
  • ✅ Resposta com body.
  • ❌ Safe Method - Pois existe a possibilidade de se alterar dados do recurso no servidor mesmo que parcialmente.
  • ❌ Idempotente - Pois a múltiplas request deste verbo afetaram o recurso com o processamento da alteração no servidor e pode afetar parcialmente o estado do recurso no servidor.
  • ⚠️ Cacheable Method - Cacheável somente se a resposta tiver definido explicitamente informações de quão recente é a informação (freshness) e um cabeçalho Content-Location for definido para o mesmo valor da URL acessada.

    Informações de freshness podem ser fornecidas através de cabeçalhos como Expires ou Cache-Control: max-age.

DELETE (RFC 7231)

Exclui um recurso que estava anteriormente salvo no servidor.

Propriedades Comuns:

  • ❌ Requisição necessita de um body.
  • ❌ Resposta com body.
  • ❌ Safe Method - Pois o recurso está afetando o estado e existência do recurso no servidor.
  • ✅ Idempotente - Pois após a exclusão do recurso, requests múltiplos não poderão exercer mais nenhum efeito no servidor já que o recurso já foi excluído.
  • ❌ Cacheable Method.

Options (RFC 7231)

Serve para recuperar informações do que é permitido fazer pela API.

Propriedades Comuns:

  • ❌ Requisição necessita de um body.
  • ❌ Resposta com body.
  • ✅ Safe Method - Pois o a consulta por si só não altera o estado do recurso no servidor.
  • ✅ Idempotente - Pois a múltiplas consultas ao recurso se manterá a mesma independente de quantas requisições repetidas sejam enviadas.
  • ❌ Cacheable Method.

Modelagem

Parâmetros de Query

Útil quando é necessário enviar dados de forma a economizar recursos trafegados na chamada da API pois os dados são passados diretamente na URL além de limitar o tamanho do response já que os dados podem ser filtrados pelos parâmetros e assim trafegar apenas o que é absolutamente necessário naquela chamada.

HATEOAS

Hypermedia As The Engine Application State (Hipermídia Como Motor De Estado Da Aplicação).

Além dos dados do recurso, é adicionado a informação sobre as operações que se pode fazer com ou a partir desse recurso.

HAL

Hypertext Application Language (Linguagem de Hipertexto da Aplicação)

Usa a mesma ideia do HATEOAS mas com uma linguagem um pouco mais desenvolvida e estruturada mas em essência tem o mesmo objetivo onde sua descrição é melhor abordada nos links a seguir.

Versionamento

Quando uma API tem sua estrutura alterada que pode acabar comprometendo as aplicações que a utilizam pode ser interessante versionar aquele endpoint de modo a que a versão da estrutura que é conhecida por todos até então possa se manter disponível até que todas as aplicações clientes sejam avisadas para atualizar para a nova estrutura.

A forma em que isso é feito basta incluir na URL do endpoint a informação de versão.

Tempo Real

Pode ser necessário que uma aplicação precise atualizar em tempo real os dados provenientes de uma API e para atualizar esses dados de maneira online é necessário o uso de alguma técnica que faça consultas a API. A seguir veremos algumas dessas técnicas e estratégias:

  • Short Polling

    Envia ao servidor requisições em um curto período de tempo para obter os dados atualizados dentro deste intervalo.

    • Vantagem
      • Simples de implementar já que basta que a aplicação crie uma rotina de enviar requisições seguidas dentro de um intervalo de tempo definido.
      • Funciona em praticamente todos os navegadores e ambientes, pois utiliza tecnologias HTTP comuns, sem a necessidade de suporte a WebSockets ou outros protocolos mais complexos.
      • O cliente tem mais controle sobre a frequência de atualização, podendo ajustar os intervalos de polling de acordo com as necessidades específicas do aplicativo ou as capacidades da rede.
    • Desvantagem
      • A atualização não é de fato em tempo real já que o dados poderá ficar desatualizado caso alguma alteração seja feita no espaço de tempo entre uma requisição e outra;
      • Dependendo do intervalo de tempo que se deseje utilizar para o envio das requisições, isso poderá adicionar ao servidor da API uma sobrecarga e causar até problemas de performance para o cliente que está fazendo as requisições e até mesmo para todos os outros.
      • Apenas comunicação unidirecional (do servidor para o cliente);
  • Long Polling

    Como uma tentativa de eliminar as desvantagens oferecidas pelo Short Polling, o Long Polling tem como estratégia enviar uma requisição para o servidor esperando que a conexão com ele se seja fechada após o recurso daquela chamada tenha sofrido alguma alteração e assim a aplicação que está fazendo a requisição passa a ter uma posição passiva já que ela apenas aguarda que o servidor tenha a iniciativa de enviar a resposta apenas após a atualização do recurso no lado do servidor.

    • Vantagem
      • Diminui a quantidade de requisições realizadas e garante que a atualização será atualizada para a aplicação cliente verdadeiramente em tempo real já que o servidor informará ao cliente imediatamente após o recurso ter sido alterado.

    • Desvantagem
      • Manter uma conexão aberta com o servidor esperando uma resposta pode ser muito custoso para ambos os lados já que mantê-la custa caro em questão de redes e processamento;
      • A implementação no lado do servidor para decidir quando enviar aos clientes a informação das alterações e de quando a sessão deve ou não ser mantida acrescenta uma maior camada de complexidade e isso pode ou não causar problemas de performance para o servidor (dependerá do quão bem implementada essa decisão foi);
      • Como o Long Polling mantém a conexão aberta por um período mais longo, pode haver casos em que a conexão é perdida devido a problemas de rede ou falhas no servidor;
      • Apenas comunicação unidirecional (do servidor para o cliente);
  • Server-Sent Events

    Uma maneira mais moderna e elegante para lidar com atualizações de forma online é o Server-Sent Events (Eventos Enviados pelo Servidor) que consiste na API fornecendo uma Event Source (Fonte de Eventos) que sempre que o cliente abrir uma conexão(que se manterá aberta) será enviado os eventos dessa fonte de forma ativa para a conexão aberta pelo cliente.

    • Vantagem
      • Ao contrário do Long Polling não é necessário enviar uma nova requisição sempre que um recurso seja alterado pois o Server-Sent Events envia sempre para as conexões abertas;
      • Caso o servidor falhe ou fique indisponível, o cliente não precisará tomar nenhuma ação pois quando o servidor for reestabelecido ele volta a enviar os eventos;
    • Desvantagem
      • Restrição do número de conexões simultâneas pelo navegador;
      • Apenas comunicação unidirecional (do servidor para o cliente);
  • WebSocket

    Uma alternativa para o Polling (Short e Long) e Server-Sent Events com a vantagem de permitir uma conexão bidirecional entre o cliente e servidor ao estabelecer um túnel de rede entre as duas partes (conexão full duplex).
